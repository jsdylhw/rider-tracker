## 骑行模拟相关分析

### 1. 当前实时数据分两层

#### 原始实时值

设备通过蓝牙上报的瞬时值，主要包括：

- 功率
- 踏频
- 心率

这些数据当前主要存在 `state.ble` 中，用于表示“设备这一刻报上来的值”。

特点：

- 更新频率由设备决定，不稳定
- 更适合做即时显示
- 不直接等于“这一秒骑行结果”

#### 1 秒采样后的骑行快照

开始骑行后，系统当前会按固定 `1s` 节奏推进一次实时骑行：

- 读取当前功率 / 踏频 / 心率
- 根据当前路线位置采样坡度
- 用物理模型推进一秒
- 生成一条新的 ride record
- 更新 summary 与训练运行态

这些结果主要落在：

- `state.liveRide.session.records`
- `state.liveRide.session.summary`
- `state.workout.runtime`

它更像系统里的“本秒权威状态”。

### 2. 当前模拟算法链路

当前实时骑行推进链路可以概括为：

1. 蓝牙设备异步上报原始值
2. 原始值写入 `ble` 状态
3. 开始骑行后按 `1s` 定时执行一次推进
4. 根据已骑行距离在路线中采样当前位置与坡度
5. 用 `power + grade + rider/bike params + drag/rolling resistance` 做一次物理推进
6. 生成新的距离、速度、海拔、位置、心率趋势
7. 写入新的 record / summary
8. 计算训练运行态并生成 trainer control 目标
9. 刷新 Dashboard、Street View、图表和训练 HUD

### 3. 当前物理模拟的核心特点

- 当前位置不是按固定段跳转，而是按累计距离在路线中取样。
- 物理推进本身支持 `dt`，理论上并不依赖 1 秒这个固定值。
- 当前的 `1s` 限制主要在调度层，而不是在物理域模型本身。

这意味着后续要提升平滑度，不一定要重写物理公式，更多是要改：

- 调度频率
- 输入数据缓存方式
- UI 刷新频率

### 4. 骑行动态渲染 与 训练 HUD 的区别

这两者不是一个模块，也不应继续混在一起。

#### 骑行动态渲染

更偏整场骑行状态，主要关心：

- 当前路线位置
- 进度
- 速度
- 坡度
- 地图/街景同步
- 路线段信息

#### 训练 HUD

更偏训练执行状态，主要关心：

- 当前训练阶段
- 当前目标功率
- 当前目标 FTP%
- 阶段剩余时间

它们共享同一份实时运行态，但回答的问题不同：

- 骑行动态渲染回答“我现在骑到哪里、速度多少、路况如何”
- 训练 HUD 回答“我这一段应该踩多少、还剩多久”

### 5. 当前耦合点

虽然已经做了一轮重构，但目前仍存在几个值得继续处理的耦合点：

- `ride-service` 仍同时承担：
  - 采样调度
  - 物理推进
  - session 更新
  - trainer 命令下发
- `dashboard-renderer` 虽然已瘦身，但仍直接协调：

- 蓝牙输入频率、仿真推进频率、UI 刷新频率目前仍未完全分离

## 后续改动建议

### 1. 先继续做结构拆分

当前建议优先级：

1. 继续拆 `trajectory renderer`
2. 再把地图/街景协调逻辑进一步收缩
3. 保持 `dashboard-renderer` 最终只做页面 orchestration

### 2. 再做实时数据处理重构

建议不要直接让位置更新完全跟随蓝牙频率，而是改成三层时钟：

#### 输入层

- 蓝牙事件驱动
- 到新数据就更新 `sensor snapshot`

#### 仿真层

- 用独立频率推进 ride engine
- 建议先尝试 `4Hz ~ 5Hz`
- 每次按最新输入值与 `dt` 推进

#### UI 层

- 图表、地图、训练 HUD、指标卡分频刷新
- 不同 UI 模块按各自需要更新

### 3. 推荐的数据分层目标

后续建议逐步形成四层：

1. `sensor-state`
   - 原始功率 / 踏频 / 心率
2. `ride-engine`
   - 物理推进、位置、速度、session record
3. `training-runtime`
   - ERG / SIM / RESISTANCE / 自定义训练目标运行态
4. `ui-view-model`
   - Dashboard / PiP / Street View / 图表所需最终展示数据

### 4. 训练与骑行展示继续解耦

已经完成的部分：

- 训练 HUD 独立
- 目标图独立
- 指标卡片独立

后续可以继续：

- 轨迹概览独立
- 地图/街景协调器独立
- Street View 左侧 HUD 独立

### 5. 仿真频率改造建议

建议分两阶段推进：

#### 第一阶段

- 保持现有算法不变
- 将 `1s tick` 改为固定高频仿真，例如 `200ms`
- 继续复用现有 `simulateStep(dt)` 能力

#### 第二阶段

- 引入更明确的输入时间戳与失效判断
- 对不同传感器做单独的时效处理
- trainer 命令下发做节流与阈值触发

### 6. 风险点

- 如果直接让位置完全跟随蓝牙包频率，不同设备会导致骑行体验差异过大。
- 如果 UI 与仿真全部同频，容易造成无意义重绘。
- 如果 renderer 继续直接读取全局 state，后续频率改造会越来越困难。

## 总结

今天这轮工作重点已经从“继续堆功能”转向“为后续骑行模拟升级打基础”：

- 图表已开始抽离
- Dashboard 已分模块
- 实时数据已开始分层

接下来最合理的路线是：

1. 继续收缩 Dashboard 与地图/街景耦合
2. 再升级实时数据处理链
3. 最后再推进更高频、更平滑的骑行模拟
