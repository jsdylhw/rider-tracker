# 0424 开发记录

## 今日改动

### 1. trainer 控制链统一为默认 best effort

- 对 `trainer-ftms` 的控制语义做了一轮收敛。
- 这次先明确一个默认原则：
  - `SIM` 默认走连续刷新
  - `固定阻力` 默认只在首次进入或目标变化时下发
  - `ERG` 默认也可以走 best effort，但不再伪装成“已确认”
- 删除了之前依赖 FTMS response 的伪确认分支，避免上层把“写入成功”误当成“trainer 已执行成功”。

这一步的核心不是增加功能，而是先把语义掰正：

- `writeValueWithResponse`
  - 表示写入特征成功
- FTMS response
  - 才是 trainer 对控制点命令的业务响应

在默认 best effort 模式下，上层不再把这两者混用。

### 2. ERG / 固定阻力 改为“只在目标变化时下发”

- 修正了 `ERG` 和固定阻力模式按 tick 重复发同一个目标的问题。
- 现在的策略是：
  - 首次进入目标阶段时下发一次
  - 目标值变化时再下发一次
  - 目标值不变则不重复下发

这更符合当前产品语义：

- 例如 `150W x 10min`
  - 阶段开始时发一次即可
  - 中间不需要每 `500ms` 重发

涉及改动：

- `src/domain/workout/erg-mode.js`
- `src/domain/workout/resistance-mode.js`
- `src/app/realtime/ride-engine.js`

### 3. trainer 命令队列和日志语义调整

- `trainer-ftms` 中控制点队列逻辑进一步清理。
- 队列过长时，被替换掉的旧命令现在会显式失败，而不是静默消失。
- 训练日志中的待发送命令显示也补强了：
  - `SIM`
  - `ERG`
  - `RESISTANCE`

这样在联调时更容易看出当前到底准备发什么控制目标。

### 4. 新增可选 ERG 确认模式

- 在默认 best effort 基础上，又补了一条“可选确认模式”。
- 这个模式只针对 `ERG`：
  - 默认关闭
  - 开启后，`ERG` 目标功率发送会等待 FTMS Control Point response
  - 收到 success 后，才把这次目标记成“已发送成功”

这样保留了两种行为：

- 快速模式
  - 更接近原先 best effort
- 确认模式
  - 更适合关键训练阶段，希望确保 trainer 已接受目标功率

涉及改动：

- `src/adapters/bluetooth/trainer-ftms.js`
- `src/adapters/bluetooth/controllable-trainer.js`
- `src/app/services/device-service.js`
- `src/app/services/ride-service.js`
- `src/domain/workout/trainer-command.js`
- `src/domain/workout/erg-mode.js`

### 5. ERG 确认模式的调度状态补齐

- 仅仅“等待回包”还不够，还需要修正调度层状态。
- 这次补了 `ride-service` 里和确认模式相关的状态流：
  - 命令确认中时，不重复下发同一条 ERG 命令
  - 收到确认成功后，才更新 `lastSentPowerWatts`
  - 如果确认失败或超时，会清掉 in-flight 状态
  - 后续 tick 会按节流策略继续重试

这样避免了一个关键问题：

- 如果开启确认模式，但第一次发送失败
- 系统不能误以为“已经发过了”
- 否则后面整段 ERG 阶段可能都不再重发

这一步属于比较关键的行为修复。

### 6. Live 页增加 ERG 确认模式开关

- 目前交互是：

  - `runtime.controlStatus` 会显示当前是：

## 当前结构判断

### 1. trainer 控制现在分成两类语义

目前控制语义已经比之前清楚很多：

- `SIM`
  - 连续控制
  - 更适合 best effort
- `ERG`
  - 阶段切换型控制
  - 可以按需启用确认模式
- `RESISTANCE`
  - 稳态目标控制
  - 当前仍默认 best effort

这比之前“所有命令都混在同一套伪确认语义里”要稳定很多。

### 2. ride-service 仍是主要调度中心

虽然今天修掉了一部分关键行为问题，但 `ride-service` 仍然偏重，继续承担：

- 实时 tick
- 命令调度
- in-flight 状态管理
- session 更新
- 导出触发

后面如果 trainer 控制继续复杂化，建议最终还是把“trainer dispatch”再单独抽一层。

### 3. trainer-ftms 现在是“双模式控制点”

现在 `trainer-ftms` 已经具备：

- 默认 best effort
- 可选 FTMS response 确认

这对后续扩展是有帮助的，后面如果还要给：

- `Request Control`
- `固定阻力`
- 某些一次性模式切换命令

增加确认能力，结构上已经有基础。

## 指标数据管理分析

### 1. 当前指标链路

目前速度、功率、心率相关指标主链路是：

1. BLE / trainer 异步上报原始值
2. 写入 `sensor-sampling`
3. `ride-service` 按 tick 取有效采样
4. `ride-engine` / `live-ride-session` 生成 `records`
5. 在 `summary` 中聚合一部分指标
6. `view-model` 再补一部分 UI 指标
7. Dashboard / PiP / 导出消费

也就是说，现在的指标并不是在一个地方统一算完，而是分散在：

- 采样层
- 会话层
- view-model 层

### 2. 当前存在的问题

#### 2.1 指标口径分散

- `averagePower`
  - 采样层有一套
  - 实时会话 summary 又有一套
- `3秒均功率`
  - 当前在 `view-model` 中按最近 3 条 record 计算
  - 不是严格意义上的 3 秒时间窗
- `NP`
  - 当前没有真实实现
  - 只有配置占位与 UI 残留

#### 2.2 UI 与分析逻辑耦合

- Dashboard view-model 现在已经开始承担“分析型指标计算”的职责
- 这会导致：
  - PiP 不易复用
  - 导出不易复用
  - 后续训练分析页不易复用

#### 2.3 后续扩展会越来越难

如果后面继续补：

- 最大速度
- 最大心率
- NP
- IF
- VI
- 时间窗功率
- 分区统计

继续分散写在不同层里，维护成本会快速上升。

### 3. 是否应该独立

结论：**应该独立。**

更准确地说，不是把整个实时骑行系统拆走，而是把：

- 指标聚合
- 指标统计
- 指标口径定义
- 指标输出对象

抽成独立的 `metrics analysis module`。

这样后续：

- Dashboard
- PiP
- 训练总结
- FIT / JSON 导出
- 后续训练分析页

都可以复用同一套指标结果。

## 指标模块设计方案

### 1. 设计目标

新模块目标不是替代实时骑行推进，而是回答：

- 当前速度是多少
- 平均速度是多少
- 最大速度是多少
- 当前功率是多少
- 平均功率是多少
- 3s / 10s 时间窗功率是多少
- NP / IF / TSS 是多少
- 当前 / 平均 / 最大心率是多少

也就是说，它是“分析层”，不是“仿真推进层”。

### 2. 推荐目录结构

建议新增：

```text
src/domain/metrics/
  index.js
  ride-metrics-service.js
  ride-metrics-schema.js
  calculators/
    speed-metrics.js
    power-metrics.js
    heart-rate-metrics.js
    cadence-metrics.js
    load-metrics.js
  windows/
    rolling-average.js
    time-window.js
```

如果希望第一版更轻量，也可以先简化成：

```text
src/domain/metrics/
  ride-metrics.js
  rolling-average.js
```

建议优先用轻量版起步，等指标种类变多再拆分。

### 3. 模块职责边界

#### `sensor-sampling`

只负责：

- 原始采样接收
- 去 stale
- 频率估计
- 采样源状态

不再承担训练统计口径定义。

#### `live-ride-session`

只负责：

- 物理推进
- 生成 `record`
- 保留最小必要 session 状态

不再持续堆加复杂分析指标。

#### `metrics module`

负责：

- 消费 `records`
- 统一计算指标
- 输出标准化 `metrics summary`

#### `view-model`

只负责：

- 格式化
- 单位转换
- 面板字段映射

#### `renderer`

只负责：

- 展示

### 4. 输入接口建议

建议新模块统一入口：

```js
buildRideMetrics({
  records,
  ftp,
  options
})
```

其中：

- `records`
  - 来自 `liveRide.session.records`
- `ftp`
  - 用于 `NP / IF / TSS`
- `options`
  - 用于时间窗、空值策略、速度口径等配置

### 5. 输出接口建议

建议输出统一结构：

```js
{
  speed: {
    currentKph,
    averageKph,
    maxKph
  },
  power: {
    currentWatts,
    averageWatts,
    maxWatts,
    rolling3sWatts,
    rolling10sWatts,
    normalizedPowerWatts,
    intensityFactor,
    variabilityIndex
  },
  heartRate: {
    currentBpm,
    averageBpm,
    maxBpm
  },
  cadence: {
    currentRpm,
    averageRpm,
    maxRpm
  },
  load: {
    tss
  }
}
```

这样可以确保：

- Dashboard 不必自己计算
- PiP 不必自己推导
- 导出层也不必重复聚合

### 6. 指标实现优先级

#### 第一批

优先实现最容易统一且收益最大的指标：

- 当前速度
- 平均速度
- 最大速度
- 当前功率
- 平均功率
- 最大功率
- 当前心率
- 平均心率
- 最大心率
- 平均踏频

#### 第二批

- 严格时间窗 `3s / 10s rolling power`
- 最大踏频

#### 第三批

- `NP`
- `IF`
- `VI`
- 基于 `NP/IF` 的更标准 `TSS`

### 7. 关键口径建议

#### 速度口径

当前主流程里速度是：

- 基于物理模型计算出的虚拟骑行速度

因此建议第一版指标模块仍使用这套速度作为主口径，避免和 trainer 上报速度混用。

后续如果需要，可以再扩展：

- `virtualSpeedKph`
- `deviceSpeedKph`

双口径输出。

#### 3 秒均功率

当前实现是最近 3 条 record 平均，这和真实 3 秒不是一回事。

建议改成：

- 以 `elapsedSeconds` 为准
- 做严格时间窗聚合

也就是：

- 取最近 `3.0s` 内样本
- 再对功率做平均

#### NP

建议按标准近似路线实现：

1. 先生成 30 秒 rolling average power
2. 每个点做四次方
3. 求平均
4. 再开四次方根

### 8. 推荐迁移顺序

建议按以下顺序迁移，而不是一次性重写：

#### 第一步：抽离分析入口

- 新增 `metrics module`
- 先只计算现有 summary 已有指标

目的：

- 不改变 UI 语义
- 先把计算入口统一

#### 第二步：替换 Dashboard 的局部计算

- 把 `view-model` 里的 `3秒均功率` 挪进 metrics module
- Dashboard 改为只读取 metrics 结果

#### 第三步：替换 summary 聚合实现

- `live-ride-session` 中的 `buildSummary`
- 改为调用 `buildRideMetrics`

这样 summary 与 view-model 就统一口径。

#### 第四步：接入导出层

- FIT / JSON 导出优先复用 metrics summary
- 避免导出层再重复算一遍平均值和最大值

#### 第五步：扩展高级指标

- `NP`
- `IF`
- `VI`
- 标准 `TSS`

### 9. 迁移后的目标结构

后续建议把指标链路收敛成：

1. `sensor-state`
   - 原始采样与有效性
2. `ride-session`
   - record 生成与实时推进
3. `metrics-analysis`
   - 指标聚合与分析
4. `ui-view-model`
   - 展示字段格式化

这比当前“summary + view-model 混合计算”更适合长期演进。

### 10. 风险点

迁移时要特别注意：

#### 口径变化导致 UI 数值变化

例如：

- `3秒均功率`
- `平均功率`
- `TSS`

在新旧口径切换时，前端显示可能会变化。

#### 旧测试需要同步更新

目前测试主要覆盖：

- 实时会话推进
- 部分 summary 聚合

但对：

- NP
- 最大心率
- 严格时间窗平均功率

覆盖还不够。

### 11. 结论

当前项目已经到了适合抽离指标分析层的阶段。

原因不是“代码不够优雅”，而是：

- 指标口径已经开始分散
- 新指标需求已经明确出现
- 后续训练分析方向也需要这一层作为基础

因此后面更合理的方向是：

- 保持 `ride engine` 专注推进
- 保持 `sensor-sampling` 专注原始采样
- 新增独立 `metrics module` 负责分析与聚合

这样后续不管是：

- Dashboard
- PiP
- 导出
- 训练总结
- 后续分析页

都能复用同一套指标结果。

### 12. 第一阶段已落地

当前已经按“轻量版 metrics module”方案完成第一阶段收口：

- 新增：
  - `src/domain/metrics/ride-metrics.js`
- `live-ride-session`
  - 不再自己维护分散的指标聚合逻辑
  - 改为调用 `buildRideMetrics`
- `simulator`
  - 也复用同一套 metrics 计算
- Dashboard view-model
  - `3秒均功率` 改为复用 metrics 结果
  - 不再自己按最近 3 条 record 临时计算

当前 `summary` 的结构已经变成：

- 兼容层
  - 保留原有 `averageSpeedKph` / `averagePower` / `estimatedTss` 等扁平字段
- 标准分析层
  - 新增 `summary.metrics`

这一步的意义是：

- 不打断旧 UI
- 不破坏旧导出逻辑
- 但新的分析口径已经有统一入口

### 13. 第二阶段已落地

第二阶段主要做了“消费侧收口”：

- PiP
  - 改为优先读取 `summary.metrics`
- 主面板
  - 平均速度、距离、平均心率、爬升、总用时、当前坡度
  - 改为优先读取 `summary.metrics`
- 首页历史摘要
  - 距离 / 用时 / 平均速度
  - 改为优先读取 `summary.metrics`
- Dashboard 进度区
  - 已优先使用统一 metrics 结果里的距离与进度
- FIT 导出
  - 平均/最大速度
  - 平均/最大心率
  - 平均/最大功率
  - 已优先消费 `summary.metrics`

也就是说，现在已经不只是“内部新增了一个 metrics 模块”，而是：

- UI 开始读它
- 导出开始读它

### 14. grade 统计已并入 metrics

原来 FIT 导出层自己有一套 `gradeStats` 汇总逻辑。

现在已经把这部分合并进 `metrics module`，输出为：

- `grade.currentPercent`
- `grade.averagePercent`
- `grade.averagePositivePercent`
- `grade.averageNegativePercent`
- `grade.maxPositivePercent`
- `grade.maxNegativePercent`

这样后续：

- UI
- FIT 导出
- 后续训练分析页

都可以直接复用统一坡度统计，不需要再各自重新汇总。

### 15. resolveRideMetrics 作为统一读取入口

为了兼容旧 session 数据，这次又补了一层读取入口：

```js
resolveRideMetrics({
  summary,
  records,
  ftp,
  options
})
```

优先级是：

1. `summary.metrics`
2. 旧 `summary` 扁平字段
3. `records` 回退重算

这意味着：

- 新数据优先走统一分析口径
- 旧数据仍可正常展示和导出

### 16. 当前已统一的指标口径

目前已经统一进入 metrics 的主要指标包括：

- 速度
  - 当前 / 平均 / 最大
- 功率
  - 当前 / 平均 / 最大
  - `3s / 10s rolling`
  - `NP / IF / VI`
- 心率
  - 当前 / 平均 / 最大
- 踏频
  - 当前 / 平均 / 最大
- 坡度
  - 当前 / 平均 / 正坡平均 / 负坡平均 / 最大正坡 / 最大负坡
- 负荷
  - `estimatedTss`

### 17. 当前仍保留的兼容层

虽然已经开始统一，但目前仍保留：

- `summary.averageSpeedKph`
- `summary.averagePower`
- `summary.averageHeartRate`
- `summary.estimatedTss`
- 以及其它旧扁平字段

保留这些字段的目的不是继续扩散旧结构，而是：

- 保证旧代码路径暂时不炸
- 保证历史 session 还能显示

长期目标仍然是：

- UI / 导出 / 后续分析页都尽量直接面向 `summary.metrics`

## 蓝牙到 FIT 的当前数据流

### 18. 蓝牙采样入口

当前蓝牙数据统一管理主要分成三层：

#### 设备接入总入口

- `src/app/services/device-service.js`

负责统一接入：

- `heart-rate-monitor`
- `controllable-trainer`

它是应用层看到的蓝牙总入口。

#### 功率来源统一抽象

- `src/adapters/bluetooth/controllable-trainer.js`

负责把：

- trainer 内置功率
- 外置功率计

统一成一套 `onData` 输出：

```js
{
  power,
  cadence,
  speedKph,
  timestamp,
  sourceType
}
```

#### 采样状态统一层

- `src/app/realtime/sensor-sampling.js`

负责：

- 接收原始样本
- 维护 freshness
- 维护平均功率
- 维护采样频率与稳定性

### 19. 实时骑行推进链路

蓝牙数据进入应用后，当前主链路是：

1. 蓝牙设备异步上报
2. `device-service` 收到 `onData`
3. 写入 `state.ble.sampling`
4. `ride-service` 每个 tick 调用 `buildEffectiveSensorSnapshot`
5. `ride-engine` 生成下一帧 `rideSnapshot`
6. `live-ride-session` 推进物理模型并生成 `record`
7. `buildRideMetrics` 基于全部 `records` 生成指标
8. 写回 `session.summary` 和 `summary.metrics`
9. UI / PiP / Dashboard / 导出消费

### 20. FIT 导出链路

FIT 导出当前链路是：

1. `export-service.downloadFit()`
2. 调用 `exportSessionAsFit(session, exportMetadata)`
3. `fit-exporter` 先解析：
   - `summary.metrics`
   - 旧 `summary`
   - `records` 回退
4. `records` 写入 FIT `RECORD` 消息
5. 统一指标写入：
   - `LAP`
   - `SESSION`
   - `ACTIVITY`
6. 返回 FIT 二进制
7. 浏览器下载或上传

### 21. 当前主口径说明

当前 FIT 中的：

- 速度
- 距离
- 累计爬升

主要仍来自实时物理推进结果，不是设备原始速度直接透传。

也就是说当前主流程是：

- BLE 主要提供：
  - 功率
  - 踏频
  - 心率
- 路线提供：
  - 坡度
  - 海拔
- 物理模型推导：
  - 速度
  - 距离
  - 爬升

这是当前“虚拟骑行”模式下的主口径。

### 22. 当前结构结论

到目前为止，数据链路已经基本形成下面的职责边界：

1. `sensor-sampling`
   - 原始样本与有效性
2. `ride-session`
   - 物理推进与 record 生成
3. `metrics-analysis`
   - 指标聚合与统计
4. `ui / fit-exporter`
   - 消费统一分析结果

这比之前：

- summary 自己算一部分
- view-model 自己补一部分
- exporter 再重算一部分

要清晰得多，也更适合作为后续训练分析能力的基础。
