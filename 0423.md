# 0423 开发记录

## 今日改动

### 1. 心率先从骑行物理推进中独立

- 将实时骑行里的心率处理，从 `cycling-model` 物理推进中拆出。
- 现在速度、距离、坡度、空气阻力、滚阻等物理推进，不再顺带计算心率。
- 实时骑行当前改为：
  - 优先使用心率带实时数据
  - 如果当前 tick 没有新的心率样本，则沿用上一条有效心率
  - 不再让物理推进过程隐式地产生“估算心率”

这样做的意义是：

- 功率/速度/坡度属于物理状态
- 心率属于生理指标流
- 这两类数据先分层，后续做分析会更清楚

### 2. 新增独立心率模型模块

- 新增独立模块：
  - `src/domain/physiology/heart-rate-model.js`

当前提供两类能力：

- `advanceLiveHeartRateState`
  - 用于实时骑行
  - 负责“新心率样本优先，否则沿用上次值”
- `estimateHeartRate`
  - 用于离线模拟或后续分析
  - 负责基于功率、时长、疲劳趋势生成估算心率

这意味着后面可以单独演进心率模型，而不用继续改物理推进公式。

### 3. 实时骑行心率语义调整

- `live-ride-session` 新增独立 `heartRateState`
- `physicsState` 不再保存心率
- `ride record` 里的 `heartRate` 现在来自独立心率状态，而不是物理公式输出

当前实时骑行里的心率语义更明确：

- 这是“当前最新有效心率展示值”
- 不是“物理推进必然产生的状态变量”

### 4. 离线模拟保留估算心率

- `simulateRide()` 仍保留平均心率、记录心率等结果
- 但实现已改为调用独立 `heart-rate-model`
- 不再依赖 `cycling-model` 内部顺带生成心率

这样后面如果要单独升级模拟心率算法，只需要改生理模型，不需要碰物理推进。

### 5. 测试补充

- 新增：
  - `tests/unit/heart-rate-model.test.js`
- 调整：
  - `tests/unit/cycling-model.test.js`
  - `tests/unit/live-ride-session.test.js`
  - `tests/test-runner.js`

当前重点覆盖：

- 实时心率优先使用传感器值
- 无新样本时沿用上一条有效心率
- 心率估算模型仍可单独验证

## 当前结构判断

这一步不是把“心率分析功能”全部做完，而是先把结构掰正：

- `cycling-model`
  - 负责物理推进
- `heart-rate-model`
  - 负责生理指标建模
- `sensor-sampling`
  - 负责原始传感器采样
- `live-ride-session`
  - 负责把实时记录组织起来

这比之前“物理推进里顺便算心率”的结构更适合后续做数据分析。

## 后续可做的数据分析方向

### 1. 保留原始事件流

建议后续增加独立事件流缓存，而不是只保留当前快照：

- `heart-rate events`
- `power events`
- `cadence events`
- `trainer control events`

每条事件保留：

- 原始时间戳
- 数据源
- 原始值
- 是否有效/是否新鲜

这样后面可以做高精度回放和分析。

### 2. 增加聚合分析层

建议在快照层之上增加一个独立 `analysis` 层，持续计算：

- 3s / 5s / 10s 平均功率
- 心率移动平均
- 功率变化率
- 心率变化率
- 心率相对功率的延迟响应
- 信号丢包率 / 抖动情况

### 3. 训练区间分析

后续可围绕训练阶段做：

- 每段平均功率
- 每段平均心率
- 目标功率偏差
- 功率达标率
- 心率漂移
- 完成度统计

这部分会非常适合配合自定义训练目标使用。

### 4. 功率-心率关系分析

这是后续很有价值的一块：

- 相同功率下心率趋势
- 心率滞后曲线
- 长时间骑行中的心率漂移
- 体能衰减趋势
- 恢复段下降速度

这会比只看单点心率更有训练价值。

### 5. trainer 与人体响应对比

如果后面保留 trainer 控制事件流，可以做：

- trainer 目标坡度变化时刻
- trainer 目标功率变化时刻
- 实际功率响应滞后
- 实际心率响应滞后
- 设备信号频率与控制响应关系

这部分也能帮助调 SIM / ERG 的控制策略。

### 6. 导出与可视化

后续可以继续做：

- 多轨时间序列图
  - 功率
  - 心率
  - 踏频
  - 速度
  - 坡度
  - trainer 目标值
- 导出分析 JSON
- 导出更细粒度 session 数据
- 后续接 Web 端分析面板

## 建议的后续结构

后续可以逐步形成：

1. `sensor event stream`
2. `sensor sampling`
3. `ride engine`
4. `ride snapshot`
5. `analysis engine`
6. `dashboard / report / export`

当前已经做到的部分：

- `sensor sampling`
- `ride engine`
- `ride snapshot`
- `heart-rate-model` 初步独立

下一步如果继续做，最值得优先推进的是：

1. 增加原始传感器事件流缓存
2. 增加独立 `analysis` 层
3. 先做功率/心率基础统计
