# 0427 开发记录

## 方向一：Rider Tracker 本地 server 整合方案

### 背景

当前项目有两条启动链路：

- `index.html` 通过 Live Server 打开，负责主界面、路线导入、模拟骑行、FIT 导出。
- `server/src/index.js` 单独启动，负责 Strava OAuth、token 保存、FIT 上传、upload status 轮询。

这个结构在开发早期可用，但后续给骑行者使用时门槛偏高。用户需要理解两个服务、端口、Strava server 地址、OAuth callback 等概念。

目标是收敛成一个本地应用入口：

```bash
npm start
```

然后只访问：

```text
http://localhost:8787
```

### 推荐目标结构

最终希望主项目变成：

```text
src/
  server/
    index.js
    strava-client.js
    token-store.js
    static-app.js
    config.js
  app/
  adapters/
  domain/
  ui/
index.html
package.json
```

其中 `src/server` 同时承担：

- serve `index.html`
- serve `src/*` 静态模块和 CSS
- 提供 Strava OAuth API
- 保存本地 Strava 凭证和 token
- 上传 FIT
- 查询 Strava upload status
- 后续接 AI 总结和活动描述写回

### 推荐迁移顺序

不要一上来同时做“移动目录 + 改 server + 改 UI + 改上传流程”。建议分三步。

#### 第一步：让现有 server 成为主入口

先保留当前目录：

```text
server/src/index.js
server/src/strava-client.js
server/src/token-store.js
```

只给 `server/src/index.js` 增加静态文件服务，让它能打开主应用：

```text
GET /        -> index.html
GET /src/*   -> src/*
GET /api/*   -> Strava API
```

验证目标：

- `http://localhost:8787` 能打开主 Rider Tracker。
- 页面里的模块 import 正常。
- `连接 Strava` 正常打开授权。
- `上传 FIT` 能使用同一个 origin 的 `/api/strava/upload-fit`。

#### 第二步：前端默认使用当前 origin

目前导出卡片里有 `Strava Server` 输入框。整合后普通用户不应该看到或填写它。

建议前端默认：

```js
window.location.origin
```

作为 Strava server URL。

UI 上可以把 `Strava Server` 变成高级设置，默认隐藏。普通流程只保留：

```text
导出 JSON
导出 FIT
连接 Strava
上传 FIT
```

点击 `上传 FIT` 后，如果未连接 Strava，再显示连接提示；如果 server 未配置 Strava client credentials，再显示配置入口。

#### 第三步：把 server 移入 `src/server`

当主流程稳定后，再做目录整理：

```text
server/src/index.js        -> src/server/index.js
server/src/strava-client.js -> src/server/strava-client.js
server/src/token-store.js   -> src/server/token-store.js
```

然后更新根目录 `package.json`：

```json
{
  "scripts": {
    "start": "node src/server/index.js",
    "test": "node tests/test-runner.js"
  }
}
```

### 本地数据与配置

server 整合后，可以顺手把本地数据能力做正规。

配置建议放：

```text
~/.config/rider-tracker/config.json
```

用于保存：

- Strava client id / secret
- access token / refresh token / expiresAt
- 用户基础偏好

数据建议放：

```text
~/.local/share/rider-tracker/
```

用于保存：

- 历史骑行 session
- 已导出的 FIT 文件
- 导入/生成的路线
- Strava upload id / activity id
- 后续 AI 分析结果

后续可设计本地 API：

```text
GET  /api/rides
GET  /api/rides/:id
POST /api/rides

GET  /api/routes
POST /api/routes

POST /api/rides/:id/export-fit
POST /api/rides/:id/upload-strava
POST /api/rides/:id/analyze
```

这样将来从本地 web app 升级到桌面 app 或云端 app 时，前端状态管理可以少改很多。

## 方向二：独立项目 - Strava 热门赛段随机路线生成器

### 核心想法

这个方向可以从 Rider Tracker 主体中独立出来，先做成一个路线实验项目。

目标不是“在 Strava 上自动创建路线”，因为 Strava 公开 API 没有创建 Route 的接口；更合适的目标是：

```text
Strava 热门赛段
+ 自己骑过/没骑过
+ 坡度/距离/热度筛选
+ 外部海拔/路由补全
-> 生成候选 GPX
-> 供 Rider Tracker 导入骑行
```

可以暂命名为：

```text
segment-route-lab
strava-route-mixer
ride-route-generator
```

### Strava Route 与 Segment 的边界

Strava 公开 API 对 Route 支持：

- 读取已有 route
- 列出用户 route
- 导出 route GPX
- 导出 route TCX

但没有官方创建 route API。

因此：

- 用户已有 Strava Route：可以通过 API 下载 GPX。
- 我们生成的新路线：可以导出 GPX 给 Rider Tracker 使用，或由用户手动上传到 Strava Routes。
- 不建议把自动建路线依赖在 Strava 产品 UI 上。

### Segment 能提供的数据

`Get Segment` 通常可以拿到：

```text
id
name
activity_type
distance
average_grade
maximum_grade
elevation_high
elevation_low
total_elevation_gain
start_latlng
end_latlng
climb_category
city / state / country
private
starred
effort_count
athlete_count
star_count
map.polyline
```

其中可以用来判断热度：

- `effort_count`：骑行/努力次数
- `athlete_count`：参与过的运动员数量
- `star_count`：收藏数量

可以用来判断路线类型：

- `average_grade`
- `maximum_grade`
- `total_elevation_gain`
- `climb_category`
- `distance`

### 海拔数据策略

Strava Segment detail 有海拔统计，但不一定直接给每个 polyline 点的海拔。

推荐标准化为：

```json
{
  "source": "strava_segment",
  "segmentId": 123,
  "name": "Example Climb",
  "distanceMeters": 5200,
  "averageGrade": 5.8,
  "maxGrade": 11.4,
  "elevationGainMeters": 310,
  "effortCount": 10000,
  "athleteCount": 2500,
  "starCount": 300,
  "start": [45.1, 6.1],
  "end": [45.2, 6.2],
  "points": [
    {
      "lat": 45.1,
      "lng": 6.1,
      "distanceMeters": 0,
      "elevationMeters": 740,
      "gradePercent": 5.2
    }
  ],
  "classification": "climb"
}
```

海拔补全优先级：

1. 尝试 Strava segment streams：
   - `latlng`
   - `distance`
   - `altitude`
   - `grade_smooth`
2. 如果 segment streams 不可用，用外部 elevation API 补点级海拔：
   - OpenTopoData
   - Open-Elevation
   - Google Elevation API
   - Mapbox Terrain
3. 如果需要连接多个 segment，中间 connector route 用 GraphHopper / openrouteservice / OSRM 生成，并请求 elevation。
4. 最后统一重算：
   - total elevation gain
   - grade per point
   - average grade
   - climb sections

### 路段分类规则

可以先用简单规则：

```js
function classifySegment(segment) {
  if (segment.average_grade >= 4 && segment.total_elevation_gain >= 80) return "climb";
  if (Math.abs(segment.average_grade) <= 1.5 && segment.distance >= 1000) return "flat";
  if (segment.average_grade <= -3) return "descent";
  return "rolling";
}
```

更细的产品分类：

- 平路：
  - average grade 在 -1.5% 到 1.5%
  - distance >= 1km
- 短坡：
  - average grade >= 4%
  - distance 0.5km - 2km
- 长爬坡：
  - average grade >= 4%
  - distance >= 3km
  - total elevation gain >= 150m
- 经典爬坡：
  - climb category > 0
  - effort_count / athlete_count / star_count 高

### 拼接多个 Segment

有两种方式。

#### 方式 A：硬拼

直接把多个 segment 的 polyline 连起来：

```text
segment A
+ segment B
+ segment C
```

适合本来连续的坡段，例如一个长爬坡被拆成多个 segment。

风险：segment 之间不连续时，GPX 会出现跳线。

#### 方式 B：智能拼接

推荐方式：

```text
segment A end_latlng
-> routing API connector
-> segment B start_latlng
```

最终路线：

```text
segment A
+ connector route
+ segment B
+ connector route
+ segment C
```

这样生成的 GPX 更适合真正骑行、街景播放和模拟。

### 随机路线生成产品形态

第一版可以先做：

```text
输入 bounds / 城市 / segment ids
-> explore Strava segments
-> get segment detail
-> 分类 flat / climb / rolling
-> 显示 effort_count / athlete_count / star_count
-> 用户勾选几个 segment
-> 补海拔
-> 导出 GPX
```

第二版再做随机组合：

```text
路线结构：
- 热身平路 5km
- 起伏路段 5km
- 经典爬坡 1 个
- 收尾平路 3km

系统：
1. 搜索候选 segment
2. 按热度和坡度分类
3. 根据用户历史过滤“骑过/没骑过”
4. 自动挑选组合
5. routing API 补连接段
6. 街景覆盖率检查
7. 导出 GPX
```

### 与 Rider Tracker 的关系

短期建议独立做实验项目，避免让主项目复杂化。

成熟后可以作为 Rider Tracker 的路线来源：

```text
路线库
- 导入 GPX
- Strava 已保存 Route
- Strava 热门 Segment
- 随机赛段路线
- 城市观光路线
```

Rider Tracker 仍然专注：

- 导入路线
- 模拟/虚拟骑行
- 街景骑行
- FIT 导出
- Strava 活动上传
- AI 骑行总结

路线生成器专注：

- 搜索 segment
- 分类 segment
- 拼接路线
- 补海拔
- 生成 GPX

## Strava 热图信息能否得到

结论：**官方公开 API 目前不能直接获取 Strava Global Heatmap 数据。**

已查到的依据：

- Strava API reference 中没有 heatmap endpoint。
- Strava Developer Hub 里官方人员曾回答 Global Heatmap API 当前不支持。
- Strava Support 说明 Global Heatmap 是聚合、去标识化、按隐私规则过滤后的社区功能。

因此不要把产品设计建立在“直接读取 Strava heatmap tile/intensity”上。

可替代方案：

1. 用 Segment 热度近似：
   - `effort_count`
   - `athlete_count`
   - `star_count`
2. 用 `explore segments` 找区域内热门段。
3. 用用户自己的 activities 生成 personal heatmap / ridden-count。
4. 用 OSM + routing API 生成连接段。
5. 如果未来需要城市级热度，需要研究 Strava Metro 或其他公开城市骑行数据，但这不是普通 Strava API 能力。

对当前路线生成器来说，最现实的热度来源是：

```text
Strava Segment popularity fields
+ user's own riding history
+ optional external route service
```
